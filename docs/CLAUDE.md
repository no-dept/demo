- 당신은 **경험 많은 시니어 Java 개발자**입니다.
- 단순히 정답 코드를 주는 것이 아니라, 왜 그렇게 설계하는 것이 좋은지 **이유와 배경을 함께 설명**합니다.

- **프로젝트 명**: 이듬해 질 녘 꽃 피는 봄
- **핵심 목표**: Spring Framework의 핵심 기능(IoC/DI, MVC)을 밑바닥부터 직접 구현해보는 리버스 엔지니어링 프로젝트입니다.

- **Build Tool**: Gradle
- **Test Framework**: JUnit 5 (JUnit Jupiter)

## 🌱 프로젝트 주차별 세부 실행 계획
Phase 1: IoC/DI 컨테이너 - '객체'를 만들고 연결하는 심장부
목표: 프레임워크가 스스로 객체를 생성하고, 객체 간의 관계를 설정해주는 핵심 원리를 코드로 증명합니다.

1주차 (9/15 ~): 어노테이션 정의 및 컴포넌트 스캔

🎯 목표: 프레임워크가 관리할 클래스(Bean)를 '찾아내는' 기반을 마련합니다.

✅ 주요 활동:

@Component, @Autowired 등 핵심 애너테이션을 @interface를 이용해 직접 정의합니다.

ComponentScanner 클래스를 구현하여, 지정된 패키지 하위의 모든 클래스 파일을 읽어오는 로직을 작성합니다.

읽어온 클래스 중에서 @Component 애너테이션이 붙은 클래스만 필터링하여 Set<Class<?>> 형태로 반환하는 기능을 완성합니다.

JUnit 테스트를 통해 스캐너가 정확하게 목표 클래스들을 찾아내는지 검증합니다.

2주차 (9/22 ~): 빈 팩토리 - '객체'를 생성하고 보관

🎯 목표: 스캔한 클래스 설계도를 바탕으로 실제 객체(Bean)를 만드는 '공장'을 구현합니다.

✅ 주요 활동:

BeanFactory 클래스를 구현하고, 내부에 Bean들을 저장할 공간(예: Map<Class<?>, Object>)을 마련합니다.

ComponentScanner가 찾아낸 클래스 목록을 받아, Java Reflection API를 사용해 각 클래스의 인스턴스(객체)를 생성합니다.

생성된 객체(Bean)를 내부 저장소에 등록하여, **싱글톤(Singleton)**으로 관리되도록 합니다.

3주차 (9/29 ~): 의존성 주입 - '객체'들을 서로 연결

🎯 목표: @Autowired를 해석하여, 특정 Bean이 필요로 하는 다른 Bean을 자동으로 연결해주는 기능을 구현합니다.

✅ 주요 활동:

BeanFactory에 등록된 Bean들을 순회하며, 생성자에 @Autowired가 있는지 확인합니다.

@Autowired가 붙은 생성자의 파라미터 타입을 확인하고, BeanFactory 저장소에서 해당 타입의 Bean을 찾아 인자로 넘겨주어 객체를 생성합니다.

이를 통해 객체 생성과 의존성 주입이 동시에 이루어지는 로직을 완성합니다. (생성자 주입)

4. 1차 통합 데모 (10/6 ~)

🎯 목표: 지금까지 만든 IoC/DI 컨테이너가 실제로 동작함을 증명합니다.

✅ 주요 활동:

간단한 PostService와 PostRepository 클래스를 만들고 각각 @Component를 붙입니다. (PostService는 생성자에서 PostRepository를 주입받도록 설계)

main 메서드에서 프레임워크를 실행시켜 두 클래스가 Bean으로 등록되고, 의존성 주입이 성공적으로 완료되는지 확인합니다.

Phase 2: MVC 프레임워크 - '웹 요청'을 받고 응답하기
목표: 직접 만든 IoC 컨테이너 위에서, 웹 브라우저의 요청을 받아 처리하고 응답하는 웹 프레임워크의 흐름을 구현합니다.

5. DispatcherServlet & 웹 어노테이션 (10/13 ~)

🎯 목표: 모든 웹 요청을 가장 앞에서 받는 단일 관문(Front Controller)을 구현합니다.

✅ 주요 활동:

@Controller, @RestController, @GetMapping 등 웹 요청 처리에 필요한 애너테이션을 정의합니다.

모든 요청(/*)을 받는 DispatcherServlet을 구현하고, 실행 시 IoC 컨테이너를 로딩하도록 연동합니다.

6. 핸들러 매핑 (10/20 ~)

🎯 목표: 들어온 요청 URL에 가장 적합한 처리 담당자(Controller 메서드)를 찾아내는 기능을 구현합니다.

✅ 주요 활동:

HandlerMapping을 구현합니다.

서버 시작 시, IoC 컨테이너에 등록된 모든 @Controller Bean을 스캔하여, @GetMapping의 URL과 해당 메서드를 Map으로 연결해 저장해 둡니다.

7. 핸들러 어댑터 (11/3 ~)

🎯 목표: 찾아낸 Controller 메서드를 실제로 '실행'시키고, 그 결과를 받아오는 어댑터를 구현합니다.

✅ 주요 활동:

HandlerAdapter를 구현합니다.

HandlerMapping이 찾아준 메서드를 Java Reflection API를 사용해 호출하고, 그 반환 값을 DispatcherServlet으로 전달합니다.

8. 2차 통합 데모 (11/10 ~)

🎯 목표: 웹 브라우저에 "Hello, World!"를 출력합니다.

✅ 주요 활동:

@RestController와 @GetMapping("/hello")를 사용한 간단한 컨트롤러를 작성합니다.

서버를 실행하고 웹 브라우저에서 localhost:8080/hello로 접속했을 때, 화면에 "Hello, World!"가 성공적으로 나타나는지 확인합니다.

Phase 3: AOP, 그리고 마무리
목표: 프록시 패턴을 이용해 공통 관심사(로깅, 트랜잭션 등)를 분리하는 AOP의 기본 개념을 구현하고, 프로젝트를 완성합니다.

9. AOP 설계 및 프록시 적용 (11/17 ~)

🎯 목표: 특정 Bean에 부가 기능을 추가할 수 있도록 프록시(Proxy)를 적용합니다.

✅ 주요 활동:

@Aspect, @Around 등 AOP 관련 애너테이션을 정의합니다.

AOP 적용 대상 Bean에 대해서는 원본 객체 대신, 부가 기능이 추가된 프록시 객체를 생성하여 Bean으로 등록하도록 BeanFactory를 수정합니다. (JDK Dynamic Proxy 또는 CGLIB)

10. AOP 어드바이스 구현 (11/24 ~)

🎯 목표: 간단한 로깅 기능을 Aspect로 구현하여 AOP의 동작을 테스트합니다.

✅ 주요 활동:

'메서드 실행 시간 측정' 로직을 담은 LoggingAspect를 구현합니다.

특정 서비스의 메서드가 호출될 때, AOP가 적용되어 실행 시간 로그가 콘솔에 정상적으로 찍히는지 확인합니다.

11. 최종 리팩토링 및 문서화 (12/1 ~)

🎯 목표: 코드의 품질을 높이고 다른 사람이 이해하기 쉽도록 프로젝트를 정리합니다.

✅ 주요 활동:

전체 코드 리뷰 및 리팩토링 (중복 제거, 가독성 향상).

README.md에 프로젝트의 구조, 실행 방법, 핵심 코드에 대한 설명을 상세히 작성합니다.

12. 프로젝트 회고 및 발표 준비 (12/8 ~)

🎯 목표: 프로젝트 경험을 내재화하고, 외부에 성과를 공유할 준비를 합니다.

✅ 주요 활동:

팀 회고(KPT: Keep, Problem, Try)를 통해 잘한 점, 아쉬운 점, 배운 점을 정리합니다.

최종 발표를 위한 데모 시나리오를 확정하고 리허설을 진행합니다.
